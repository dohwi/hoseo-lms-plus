# GitHub Workflow Korean

## What this skill does

- GitHub 작업 전반을 한국어로 정리하고 안내한다.
- 스킬 파일 경로만 호출되어도 가능한 범위의 GitHub 워크플로를 자동으로 끝까지 진행한다.
- 이슈, 브랜치, 커밋, 풀 리퀘스트, 리뷰 대응, 병합 전 체크리스트 작성을 돕는다.
- 저장소 작업 시 github-flow 방식을 기본 원칙으로 삼는다.
- 기본 브랜치 명칭은 예외 없이 `main`을 사용한다.
- 병합 방식은 squash merge를 기준으로 설명하고 제안한다.
- 커밋 메시지는 `fix`, `feature` 같은 영어 접두사를 사용하고, 뒤의 설명은 한글로 작성하는 규칙을 따른다.

## When to use

다음과 같은 요청이 있을 때 이 스킬을 사용한다.

- GitHub 이슈를 만들거나 다듬어야 할 때
- 브랜치 이름을 정해야 할 때
- 커밋 메시지를 작성하거나 정리해야 할 때
- 풀 리퀘스트 제목과 본문을 작성해야 할 때
- 코드 리뷰 의견에 대한 답변을 한국어로 정리해야 할 때
- 병합 또는 배포 전에 체크리스트를 준비해야 할 때
- `@skills/github-workflow-ko/SKILL.md`처럼 스킬 파일 경로만 호출하고, 나머지 GitHub 작업은 자동 진행되길 원할 때

## Default behavior

- 특별한 제약이 없으면 설명만 하지 말고 실제 작업을 수행한다.
- 저장소 상태를 확인한 뒤, 필요한 브랜치 작업, 커밋, 푸시, PR 생성까지 가능한 범위를 연속으로 진행한다.
- 사용자가 단순히 스킬을 호출한 경우에도 현재 변경사항이 있으면 PR 생성까지를 기본 목표로 삼는다.
- 사용자에게 중간 단계마다 허락을 구하지 않는다. 다만 파괴적이거나 되돌리기 어려운 작업은 예외다.
- PR 생성이 완료되면 결과를 먼저 공유하고, 이어서 즉시 squash merge를 진행할지 한 번만 물어본다.
- 병합 여부를 묻기 전에는 자동 병합하지 않는다.
- 기본 브랜치를 설명하거나 명령을 제안할 때는 항상 `main`만 사용하고 `master`는 대안으로도 제시하지 않는다.

## Execution policy

1. 먼저 현재 브랜치, 변경사항, 원격 추적 상태를 확인한다.
2. 기본 브랜치는 항상 `main`으로 간주하고, 작업 브랜치가 없으면 `main`에서 github-flow 기준에 맞는 브랜치를 만든다.
3. 커밋이 필요하면 `type: 한글 설명` 형식으로 커밋 메시지를 작성해 커밋한다.
4. 원격 브랜치가 없거나 최신 상태가 아니면 필요한 푸시를 진행한다.
5. PR이 없으면 한국어 제목과 본문으로 PR을 생성한다.
6. PR URL과 핵심 변경사항을 사용자에게 간단히 공유한다.
7. 마지막으로 `지금 바로 squash merge할까요?`처럼 즉시 병합 여부를 묻는다.

## Guardrails

- 기본 브랜치 이름은 항상 `main`으로 취급하며, `master`를 기본 브랜치 후보로 사용하거나 안내하지 않는다.
- 사용자가 명시적으로 원하지 않았다면 `main`에 직접 커밋하지 않는다.
- 병합 전에는 PR 상태, 리뷰, 체크 상태를 확인 가능한 범위에서 확인한다.
- 체크 실패, 리뷰 미완료, 권한 부족처럼 병합 조건이 맞지 않으면 사유를 설명하고 다음 액션만 제안한다.
- 사용자가 병합을 승인하면 squash merge를 우선 사용한다.
- 사용자가 병합을 승인하지 않으면 PR 링크와 현재 상태만 남기고 작업을 멈춘다.

## Workflow principles

### 1. GitHub Flow

- 기본 브랜치는 항상 배포 가능한 상태로 유지한다.
- 기본 브랜치 이름은 `main`으로 고정한다.
- 작업이 필요하면 `main`에서 새 작업 브랜치를 만든다.
- 작업 단위는 가능한 한 작고 명확하게 유지한다.
- 작업이 끝나면 풀 리퀘스트를 통해 변경사항을 검토한다.
- 리뷰와 확인이 끝나면 `main`으로 병합한다.

### 2. Merge strategy

- 기본 병합 방식은 squash merge를 사용한다.
- 하나의 작업 브랜치에서 발생한 여러 커밋은 병합 시 하나의 커밋으로 정리한다.
- 최종 병합 커밋 메시지는 작업의 목적이 잘 드러나도록 간결하게 작성한다.
- 커밋 히스토리는 읽기 쉽고 추적 가능하도록 유지한다.

### 3. Commit message convention

- 커밋 메시지는 영어 접두사 + 한글 설명 형식을 사용한다.
- 접두사는 작업 성격이 드러나도록 짧고 명확하게 선택한다.
- 설명은 무엇을 바꿨는지보다 왜 바꿨는지가 드러나도록 작성한다.
- 불필요하게 긴 문장보다는 짧고 의도가 분명한 표현을 사용한다.

예시:

- `fix: 로그인 실패 시 에러 문구가 보이지 않던 문제 수정`
- `feature: 사용자 프로필 편집 화면 추가`
- `fix: 만료된 토큰 처리 로직 예외 케이스 보완`

## Recommended output guidelines

### Issues

- 배경, 목표, 작업 범위, 확인 방법을 포함한다.
- 체크리스트가 있다면 바로 실행 가능한 항목으로 작성한다.
- 설명은 한국어로 간결하게 정리한다.

### Branches

- 브랜치 이름은 짧고 명확하게 작성한다.
- 가능하면 kebab-case를 사용한다.
- 작업 목적이 드러나는 접두사를 붙인다.

예시:

- `feature/profile-edit`
- `fix/login-error-message`

### Commits

- 형식은 `type: 한글 설명`을 사용한다.
- 접두사는 영어로 유지한다.
- 설명은 한글로 작성한다.
- 한 커밋에는 하나의 의도가 드러나도록 구성한다.

자주 쓰는 접두사 예시:

- `feature`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `refactor`: 동작 변화 없는 구조 개선
- `test`: 테스트 추가 또는 수정
- `chore`: 기타 유지보수 작업

### Pull requests

- 제목은 변경 목적이 한눈에 보이도록 작성한다.
- 본문에는 요약, 주요 변경사항, 확인 방법을 포함한다.
- 설명은 한국어로 작성하되, 브랜치나 커밋 규칙은 기존 규칙을 따른다.
- 병합은 squash merge를 기준으로 준비한다.
- PR 생성 후에는 결과를 공유한 다음 즉시 squash merge 진행 여부를 질문한다.

### Reviews

- 리뷰 의견은 한국어로 명확하게 설명한다.
- 수정 요청 시 문제점과 이유를 함께 적는다.
- 승인 전 확인해야 할 사항이 있으면 체크리스트로 정리한다.

## Working process

1. 현재 요청이 이슈, 브랜치, 커밋, PR, 리뷰, 릴리스 준비 중 무엇인지 식별한다.
2. 저장소 컨텍스트가 있으면 현재 변경사항, 브랜치 상태, 원격 추적 상태를 먼저 확인한다.
3. 특별한 제한이 없으면 안내만 하지 말고 실제 GitHub 작업을 끝까지 진행한다.
4. 브랜치 전략은 기본 브랜치를 `main`으로 고정한 github-flow를 기준으로 적용한다.
5. 커밋 메시지는 영어 접두사와 한글 설명 규칙을 유지한다.
6. PR은 한국어 제목과 본문으로 작성하고 squash merge를 전제로 준비한다.
7. PR 생성이 끝나면 링크와 요약을 공유한 뒤 즉시 병합할지 물어본다.
8. 결과물은 짧고 복사해서 바로 쓸 수 있게 작성한다.
